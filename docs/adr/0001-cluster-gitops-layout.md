# ADR 0001 — Per-cluster GitOps layout: layered ports + functional bundles

- **Status:** Accepted
- **Date:** 2026-06-22 (accepted 2026-06-27, impl WP-050)
- **Scope:** `kubernetes/` Flux layout across all clusters (single repo)
- **Supersedes:** the current `infrastructure/{crds,controllers}` shared-bundle layout

## Context

The current layout installs CRDs and controllers as **single shared bundles** applied
identically to every cluster:

- each cluster's `infra-crds.yaml` / `infra-controllers.yaml` Flux Kustomization points at
  the same shared path (`./kubernetes/infrastructure/{crds,controllers}`);
- those shared kustomizations list a fixed set (gateway-api; cert-manager + NGF + reloader);
- there is **no per-cluster selection** — a cluster gets the whole bundle or nothing, and
  every cluster runs the same component versions.

This blocks real needs:

- a Vault-only host (`ips-ger-vps-2`) that should not pull VPN workloads, and later
  consumer clusters that want ESO while the Vault host does not;
- running **different component versions per cluster** (canary / staged rollout);
- keeping a base free of things that aren't required for it to function (e.g. cert-manager
  must not drag a `ClusterIssuer` it doesn't need).

We evaluated the two-repo `head`/`infra` fleet pattern (brainfair) and decided **not** to
split repos (overkill for a 3-cluster, single-owner homelab — see Alternatives). The
selection ergonomics it offers are achievable in one repo.

## Decision

Adopt a **single-repo** layout that combines:

1. **Layered ports** for hard ordering (the finite shared platform).
2. **Functional bundles** for capability composition.
3. **Per-cluster variables** (`cluster-vars` + Flux `postBuild.substituteFrom`) for
   parameterization and per-cluster component versions.
4. **À-la-carte** includes as the escape hatch for one-offs.
5. **Per-app `dependsOn` (DAG)** for apps that need ordering — *not* more global layers.

Mental model: **a cluster is a set of capabilities** (`platform`, `public-tls`,
`vpn-stack`, …) — like Ansible roles — selected per cluster, parameterized by that
cluster's vars, ordered by a small layer skeleton plus per-app dependencies.

### Catalogs (pure, no cluster names, no hardcoded versions)

```
kubernetes/
├── crds/                       # CRD groups, hoisted, version via ${...}
│   ├── gateway-api/            #   ref.tag: ${gateway_api_version}
│   └── cert-manager/           #   cert-manager CRDs, pin ${cert_manager_version}
├── infra/
│   ├── base/                   # infra building blocks; HelmRelease version: ${...}
│   │   ├── cert-manager/       #   crds.enabled:false, ISSUER-AGNOSTIC
│   │   ├── nginx-gateway-fabric/
│   │   ├── reloader/
│   │   └── letsencrypt-issuers/#   ClusterIssuers (opt-in; email/solver via ${...})
│   └── bundles/
│       ├── platform/           #   resources: [cert-manager, nginx-gateway-fabric, reloader]
│       └── public-tls/         #   resources: [letsencrypt-issuers] (runtime-needs platform)
└── apps/
    ├── base/                   # 3x-ui, anylink, gost, vault; version: ${...}
    └── bundles/
        └── vpn-stack/          #   resources: [3x-ui, anylink, gost]
```

A **bundle** is just a `kustomization.yaml` with a `resources:` list — a curated group of
bases. Bundles must be **non-overlapping** (each component in exactly one bundle) or
Kustomize errors on duplicate resources. Some bundles have an **implicit runtime
dependency** (`public-tls` needs cert-manager from `platform`) — document it.

### Per-cluster entrypoint: 4 ordered layers

```
clusters/<cluster>/
├── flux-system/                # generated by `flux bootstrap`
├── cluster-vars.yaml           # ConfigMap (flux-system ns): subdomain + ALL versions for this cluster
├── crds.yaml                   # Flux KS → ./crds                         wait:true
├── controllers.yaml            # Flux KS → ./infra/controllers  dependsOn:crds         wait:true
├── config.yaml                 # Flux KS → ./infra/config       dependsOn:controllers  (issuers/gateway/cert)
├── apps.yaml                   # Flux KS → ./apps               dependsOn:config
├── infra/
│   ├── controllers/kustomization.yaml   # resources: [infra/bundles/platform]
│   └── config/
│       ├── kustomization.yaml           # resources: [infra/bundles/public-tls, gateway.yaml, certificate.yaml]
│       ├── gateway.yaml                 # listeners; host uses ${cluster_subdomain}
│       └── certificate.yaml
└── apps/
    ├── kustomization.yaml               # resources: [apps/bundles/vpn-stack | apps/base/vault] + customs
    └── <app>/...                        # per-app customs (httproute, sops secrets)
```

Each layer Flux Kustomization carries
`postBuild.substituteFrom: [{kind: ConfigMap, name: cluster-vars}]`.

**Ordering (the 4 layers solve the async-readiness problem):**
`crds (wait) → controllers (wait, dependsOn crds) → config (dependsOn controllers) → apps (dependsOn config)`.
`wait: true` makes a Kustomization Ready only when its resources are healthy, so
cert-manager is up *before* `ClusterIssuer`/`Certificate` apply — no NotReady window.

### Versions: per-cluster, never floating

Bases **omit hardcoded versions** and use `${...}` placeholders; each cluster pins **all**
its versions in `cluster-vars`:

```yaml
apiVersion: v1
kind: ConfigMap
metadata: {name: cluster-vars, namespace: flux-system}
data:
  cluster_subdomain: "vps-2.ger.ips.sanch.pet"
  cert_manager_version: "v1.21.0"   # this cluster is the canary
  ngf_version: "1.6.1"
  reloader_version: "1.x"
  vault_chart_version: "0.30.0"     # MUST be pinned — never latest (Vault has migration steps)
```

> **Never leave a version unset** (Flux would float to latest = non-reproducible, surprise
> upgrades). Per-cluster pinning is reproducible *and* gives canary/staged rollout.

Staged rollout: bump the canary cluster's `cluster-vars` → validate → bump the others.
Automate base/version bumps with Renovate.

### Apps: single layer now, per-app DAG when needed

Today's apps (3x-ui, anylink, gost, vault) ship **no operators/CRDs**, so a single `apps`
layer is sufficient. **Do not** add global layers for app-level ordering. When an app first
brings its own operator + CR, convert **only that app** to its own `ks.yaml` Flux
Kustomization with `dependsOn` (onedr0p style):

```yaml
# apps/cnpg-operator/ks.yaml — installs the operator
kind: Kustomization
metadata: {name: cnpg-operator, namespace: flux-system}
spec: { path: ./kubernetes/apps/base/cnpg-operator, wait: true, ... }
---
# apps/myapp-db/ks.yaml — the CR, ordered after the operator
kind: Kustomization
metadata: {name: myapp-db, namespace: flux-system}
spec: { dependsOn: [{name: cnpg-operator}], path: ./kubernetes/apps/base/myapp-db, ... }
```

Ordering lives where the dependency is real, not in an ever-growing layer stack.

## Target layout for the current 3 clusters

| Cluster | Role | Capabilities |
|---|---|---|
| `ips-ger-vps` | VPN node | `platform` + `public-tls` + `vpn-stack` (+ gateway/cert customs) |
| `ips-usa-vps-2` | VPN node | `platform` + `public-tls` + `vpn-stack` (+ gateway/cert customs) |
| `ips-ger-vps-2` | Vault host | `platform` + `public-tls` + à-la-carte `vault` |

Gateway listeners differ per cluster (VPN: `sub-https:8443`, Vault: `vault-https:443`), so
`gateway.yaml`/`certificate.yaml` stay per-cluster customs; only hostnames are
parameterized via `${cluster_subdomain}`.

## Consequences

### Positive

- **Readable**: a cluster declares its capability set; self-documenting.
- **DRY**: each block/bundle defined once (`platform` ×3, `public-tls` ×3, `vpn-stack` ×2).
- **Clean catalogs**: `crds/infra/apps` hold only building blocks; no nested `clusters/`
  inside layers; `clusters/` is the canonical top-level Flux dir only.
- **Granular**: a cluster takes only what it needs (Vault host: no VPN stack); à-la-carte
  covers one-offs.
- **Deterministic ordering** via the 4 layers + `wait`/`dependsOn`; no async issuer window.
- **Per-cluster versions** → canary / staged rollout — impossible with the old shared bundle.
- **Issuer-agnostic cert-manager**; apps may bring their own issuers.
- **Extensible**: new capability = new bundle; new cluster = pick bundles. Mirrors
  platform-engineering practice.

### Negative / costs

- **Bundle-partitioning discipline**: one component per bundle (else duplicate-resource
  errors); implicit inter-bundle deps (`public-tls`→`platform`) must be documented.
- **Bump-all touches N `cluster-vars`** (vs one base edit) — mitigate with Renovate.
- **CRD hoist cost**: cert-manager (and future ESO) CRDs managed explicitly in `crds/`
  (`crds.enabled:false`); CRD version must track the chart version on upgrades.
- **Slightly ahead of need** for 3 clusters (à-la-carte alone would also work) — accepted
  as an investment in structure + skill.
- **Migration**: a one-time refactor of all 3 clusters (see below), tested via
  `flux reconcile`, with no downtime to running VPN workloads.

## Migration plan (high level)

1. Build the new catalogs (`crds/`, `infra/{base,bundles}`, `apps/{base,bundles}`) from the
   current `infrastructure/` + `apps/` contents; parameterize versions as `${...}`.
2. Add `cluster-vars` per cluster (subdomain + pinned versions matching current state).
3. Rewrite each cluster entrypoint to the 4-layer form with `substituteFrom`.
4. Migrate cluster-by-cluster; `flux reconcile kustomization --with-source` and verify
   health before the next cluster. Keep changes additive where possible; rely on `prune`
   only after the new tree reconciles cleanly.
5. Remove the old `infrastructure/clusters/<c>` + `apps/clusters/<c>` once migrated.

This is its **own work product** (separate from the Vault scaffold). Sequencing options:
(a) refactor first, Vault lands in the new layout; or (b) land the Vault scaffold on the
current layout now (branch `feat/vault-ger-scaffold`), migrate all three together later.

## Alternatives considered

- **brainfair two/three-repo `head`/`infra`/`crds`** — fleet pattern (separate selection,
  catalog, promotion; `postBuild.substitute` for per-cluster values). Rejected: 3 repos +
  cross-repo sources + promotion machinery is overkill for 3 single-owner clusters; the
  same à-la-carte selection is available in one repo. Borrowed: `postBuild.substitute`.
- **brainfair profile-bundle (`docker-flex`, `hetznerk3s-minimal`)** — one big bundle per
  cluster *archetype* + delete-patches to subtract. Pays off with many clusters per few
  archetypes; rejected here (clusters are few and mostly unique; the Vault host would be a
  one-member archetype; additive composition avoids delete-patches).
- **Kustomize `kind: Component`** — kept in reserve. Components are for reusable
  *modifications that patch* existing resources; pure include/exclude is `resources:`
  selection, and per-cluster one-off tweaks are `patches:`. No dedicated `components/` layer.
- **Floating versions (omit `version`)** — rejected: non-reproducible, surprise upgrades;
  unacceptable for Vault. Versions are pinned per-cluster instead.
- **More global layers (5th, 6th)** for app-level CRD/controller ordering — rejected:
  dependencies are a DAG, not a line; use per-app `dependsOn` instead.

## Open questions

- Inline `substitute` vs `substituteFrom` ConfigMap for `cluster-vars` (one file per
  cluster vs values spread across the 4 layer KS). Leaning `substituteFrom` for the
  "all of a cluster's versions in one place" readability.
- Whether the shared VPN gateway (identical-modulo-host on ger/usa) is worth a
  parameterized `infra/bundles/vpn-gateway` vs staying a per-cluster custom.
- Wildcard cert per cluster (one `*.<subdomain>` cert via DNS-01) vs current per-hostname
  HTTP-01 Certificates — only if a DNS-01 solver is wired.

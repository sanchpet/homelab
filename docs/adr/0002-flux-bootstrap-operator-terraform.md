# ADR 0002 — Flux bootstrap via Flux Operator + Terraform (not the CLI)

- **Status:** Accepted
- **Date:** 2026-06-28 (impl WP-042, canary on `ips-ger-vps-2`)
- **Scope:** how Flux gets *installed* into a cluster (the bootstrap seed) — orthogonal to
  the manifest layout of ADR-0001 (what Flux reconciles)
- **Supersedes (for new clusters):** the imperative `flux bootstrap` CLI step

## Context

Bootstrap is a **seed problem**: GitOps means "git is the source of truth, a controller
reconciles the cluster to it" — but *something* must install that controller first, and
that something is, by definition, not yet GitOps-managed. You cannot bootstrap GitOps with
GitOps (infinite regress); there is always one first imperative act. The real quality axis
is therefore not "imperative vs declarative" (the seed is always imperative) but **how
cleanly ownership hands off** to Flux afterwards — does the seeding tool stop touching the
system, or stay on the hook for steady-state reconciliation forever?

Current state: `ips-ger-vps` and `ips-usa-vps-2` were bootstrapped with the **`flux
bootstrap` CLI** (commits a `flux-system/` dir with `gotk-components`/`gotk-sync` to git;
the operator runs a one-shot install from the laptop). A new dedicated node `ips-ger-vps-2`
(Vault host, WP-042) is being provisioned. Goals for this node: a **declarative** bootstrap
(no laptop CLI step), consistent with the Terraform-leaning stack (a `vault` TF provider is
planned, WP-041/043), and adopting the **current SoTA** method as a skill investment
(P3/P4 career signal).

## Decision

Bootstrap Flux on the new cluster with the **Flux Operator + `FluxInstance` CRD**, seeded by
the **`terraform-kubernetes-flux-operator-bootstrap`** module (ControlPlane). Terraform owns
**only the ephemeral seed** — a namespace, temporary RBAC, and a Job that applies the Flux
Operator and the `FluxInstance`; Flux then adopts those resources and Terraform stops
touching them. The `FluxInstance` declares the desired Flux (versions, the git sync pointing
at `kubernetes/clusters/ips-ger-vps-2`), so Flux's own lifecycle is GitOps/declarative, not
a CLI re-run.

**Canary discipline (per WP-050):** the new greenfield node validates the method first
(zero migration risk). If it pans out, migrate `ips-ger-vps` and `ips-usa-vps-2` off the
CLI bootstrap in a separate work product. A temporary two-method state (new node =
operator, old two = CLI) is accepted for the canary window.

**Auth:** none for git. The homelab repo is **public** and core Flux is **read-only**
(no image-automation components → no write-back to git), so Flux pulls over anonymous
HTTPS — no pullSecret, no token, no GitHub App. (In-repo secrets stay SOPS-encrypted,
decrypted in-cluster via the separate sops-age key.) A GitHub App would be the SoTA choice
*if* auth were ever needed — the repo going private, or adding image-automation write-back —
but that is YAGNI here.

Decision passed ArchGate (profile below). The chosen method is the Flux team's own
recommended 2026 approach ("Bootstrapping Flux with Terraform, the right way").

## Implications for the manifest layout (ADR-0001)

The Kustomize overlays Flux reconciles are **unchanged** — the per-cluster 4-layer
entrypoint, bundles, and `apps/base/vault` from WP-042 all stay. What changes is the seed:

- **No committed `flux-system/`** generated by the CLI. The cluster root `kustomization.yaml`
  must **drop `flux-system`** from its `resources:`; the `GitRepository` + sync `Kustomization`
  are created by the `FluxInstance` instead.
- A **`terraform/`** entry provisions the operator bootstrap for this cluster.

(PR #51 was authored against the CLI assumption; it is adjusted for this ADR before merge.)

## Profile (ArchGate ЭМОГССБ — critical: Modernity + Learnability)

| Trait | Status | Note |
|---|---|---|
| Evolvability | ✅ | clean ownership handoff; `FluxInstance` = declarative Flux lifecycle; temporary dual-method is a planned migration state |
| Scalability | ✅ | operator/FluxInstance designed for fleets; mild overkill at 3 nodes, no harm |
| Learnability ⭐ | ✅ | real learning curve, but that **is** the goal; first-principles understanding + greenfield sandbox |
| Generativity | ✅ | reusable pattern: TF-managed bootstrap, GitHub App auth, migration path for the other two |
| Speed | ✅ | one-time seed; +1 small operator pod; reconcile unaffected |
| Modernity ⭐ | ✅✅ | the SoTA (Flux 2026 blog + ControlPlane module); old `terraform-provider-flux` deprecated |
| Security | ✅ | public repo + read-only Flux → **no git secret at all**; in-repo secrets stay SOPS-encrypted (sops-age, separate); ephemeral temp-RBAC; +1 pod surface |

Verdict: clean profile (no ❌, no ⚠️), both criticals green → passes the conjunctive veto.

## Consequences

### Positive
- Declarative bootstrap in IaC; no laptop CLI step.
- Clean provisioning→reconcile handoff (Terraform not on the hook for steady state).
- Flux lifecycle (versions/config) becomes GitOps-managed via `FluxInstance`.
- SoTA method; reusable for the fleet.
- **Drops the CLI's bootstrap-time git write** (the `flux bootstrap` CLI committed
  `gotk-components`/`gotk-sync` to the repo, needing a repo-write PAT). The operator installs
  Flux from an OCI artifact per the FluxInstance — nothing is written back to git, so no
  write token. Fewer privileges, smaller surface.

### Negative / costs
- Two bootstrap methods coexist during the canary (old two on CLI) — resolved by a
  follow-up migration WP.
- One always-on operator pod per cluster.
- Learning curve (operator, CRD, TF module, GitHub App) — accepted as the goal.
- PR #51 needs a bootstrap-layer adjustment before merge.

## Alternatives considered

- **`flux bootstrap` CLI (status quo)** — consistent with the other two nodes, but imperative
  (laptop seed) and teaches nothing new; rejected for the new node (does not advance the
  critical Modernity/Learnability goals).
- **`terraform-provider-flux` (`flux_bootstrap_git`)** — declarative, but **deprecated**: it
  keeps Terraform on the hook for steady-state reconcile (ownership war with Flux); the Flux
  team recommends migrating off it. Rejected on the critical Modernity axis (ArchGate veto).

## Open questions

- Git auth is **not needed now** (public repo, read-only Flux). Revisit only if the repo
  goes private or image-automation (git write-back) is added — then a GitHub App is the
  SoTA auth.
- Where the bootstrap Terraform lives (`terraform/live/flux-<cluster>/`) and how it composes
  with the planned `vault` provider config.
- Whether to keep the `FluxInstance` manifest in git (operator reconciles it) vs TF-only.
- Migration sequencing of the existing two nodes (separate WP) — likely one at a time with
  `flux reconcile` verification, same canary discipline.
